不同节点间差异显著：aiops-k8s-04 和 aiops-k8s-07 明显承压，而 aiops-k8s-01、aiops-k8s-05 则负载减轻，暗示服务调度或流量分布发生偏移。

#### 2. Service级别现象观察

多个微服务在故障期间表现出一致性的尾部延迟恶化，具有服务级共性：

- **emailservice**（Pods -0, -1, -2）整体 rrt 99分位数显著升高（如 emailservice-1 从 2727.2 → 5512.42），尽管各 Pod 的网络流量（`pod_network_receive_bytes`）略有下降或持平，但 `pod_memory_working_set_bytes` 99分位数普遍上升（如 emailservice-0 从 784435.44 → 1105394.11），表明内存压力可能与延迟恶化相关。
- **currencyservice** 的三个 Pod 均出现 rrt 99分位数上升，其中 currencyservice-1（部署于 aiops-k8s-05 和 aiops-k8s-07）增幅最显著（1948.04 → 2421.58），且其 `pod_memory_working_set_bytes` 中位数翻倍（883.92 → 1465.24），提示服务内部处理效率下降或资源竞争加剧。
- **paymentservice** 表现不一致：paymentservice-0（aiops-k8s-08）rrt 99分位数从 2874.8 升至 3602.27，而 paymentservice-2（aiops-k8s-04）反而下降，说明问题非服务全局，可能与所在节点状态相关。
- **adservice** 三个 Pod 的请求量（`pod_network_receive_bytes`）均大幅下降（如 adservice-2 从 515.62 → 107.29），且 `request` 指标99分位数降至固定值 48.0，反映上游调用骤减，属服务级流量模式变化。

#### 3. Pod级别现象观察

部分 Pod 表现出孤立异常，且与所在节点高度相关：

- **shippingservice-0**（部署于 aiops-k8s-01）rrt 99分位数从 1867.95 激增至 5335.73，而同服务的 shippingservice-1（aiops-k8s-07）和 shippingservice-2（aiops-k8s-03）无显著延迟上升，表明问题局限于该 Pod。其所在节点 aiops-k8s-01 内存压力减轻，排除节点资源瓶颈，可能为 Pod 内部状态异常。
- **cartservice-2**（部署于 aiops-k8s-08）在故障期间出现 timeout 99分位数为 1.0（正常为 0.0），而 cartservice-0（aiops-k8s-07）和 cartservice-1（aiops-k8s-05）无此现象，且其 `pod_memory_working_set_bytes` 99分位数从 1.53M 升至 1.95M，略高于其他实例，可能因内存或 GC 导致偶发超时。
- **checkoutservice-2**（aiops-k8s-05）维持高延迟（rrt 99分位数约 8500ms），但未进一步恶化，其 `pod_network_*` 指标稳定，所在节点内存使用率下降，表明其高延迟为固有状态而非新发故障。
- 异常 Pod 分布广泛：shippingservice-0（aiops-k8s-01）、cartservice-2（aiops-k8s-08）、checkoutservice-2（aiops-k8s-05）分别位于不同节点，未集中于单一故障节点，说明多数异常为 Pod 或服务逻辑层面，非节点基础设施全局失效。

#### 综合现象描述

系统在故障期间呈现“局部高延迟 + 部分服务流量骤降”的复合现象。Node 层面，aiops-k8s-04 和 aiops-k8s-07 出现显著内存压力（`node_memory_usage_rate` 上升、`node_memory_MemAvailable_bytes` 下降），其中 aiops-k8s-04 还伴随磁盘写延迟（`node_disk_write_time_seconds_total`）激增。Service 层面，emailservice 和 currencyservice 普遍出现尾部延迟恶化，adservice 请求量锐减。Pod 层面，shippingservice-0 和 cartservice-2 表现孤立异常，分别体现高延迟和超时，但分布于不同节点，表明问题根源更可能在应用层或调度配置，而非底层基础设施大面积故障。整体来看，异常现象以服务级和 Pod 级为主，节点级异常仅限个别高负载节点。

### 系统指标综合分析结果:
### 全局现象总结分析

#### 1. Node级别现象观察

多个节点在故障期间表现出显著的资源使用变化，尤其集中在内存和磁盘写入方面：

- **aiops-k8s-04** 节点的 `node_memory_usage_rate` 中位数从 41.50 上升至 57.16，99分位数从 50.17 升至 59.68；同时 `node_memory_MemAvailable_bytes` 中位数从 9.28GB 下降至 6.66GB，表明该节点内存压力显著增加。
- **aiops-k8s-07** 节点的 `node_memory_usage_rate` 中位数从 66.27 上升至 74.78，99分位数从 71.94 升至 77.94，为所有节点中内存使用率最高且增幅最明显者，显示严重内存紧张。
- **aiops-k8s-01** 和 **aiops-k8s-05** 节点则呈现相反趋势：`node_memory_usage_rate` 中位数分别从 40.12 降至 31.51、41.89 降至 36.58，可用内存增加，表明负载可能被迁移或请求减少。
- 磁盘写入方面，**aiops-k8s-04** 的 `node_disk_write_time_seconds_total` 99分位数从 0.02 骤增至 0.27，是唯一出现显著磁盘写延迟上升的节点，可能与高内存压力并发导致 I/O 性能下降有关。
- 其他节点如 aiops-k8s-03、aiops-k8s-06、aiops-k8s-08 的 `node_cpu_usage_rate` 未提供数据，但其内存和网络指标波动较小，整体表现相对稳定。

不同节点间差异显著：aiops-k8s-04 和 aiops-k8s-07 明显承压，而 aiops-k8s-01、aiops-k8s-05 则负载减轻，暗示服务调度或流量分布发生偏移。

#### 2. Service级别现象观察

多个微服务在故障期间表现出一致性的尾部延迟恶化，具有服务级共性：

- **emailservice**（Pods -0, -1, -2）整体 rrt 99分位数显著升高（如 emailservice-1 从 2727.2 → 5512.42），尽管各 Pod 的网络流量（`pod_network_receive_bytes`）略有下降或持平，但 `pod_memory_working_set_bytes` 99分位数普遍上升（如 emailservice-0 从 784435.44 → 1105394.11），表明内存压力可能与延迟恶化相关。
- **currencyservice** 的三个 Pod 均出现 rrt 99分位数上升，其中 currencyservice-1（部署于 aiops-k8s-05 和 aiops-k8s-07）增幅最显著（1948.04 → 2421.58），且其 `pod_memory_working_set_bytes` 中位数翻倍（883.92 → 1465.24），提示服务内部处理效率下降或资源竞争加剧。
- **paymentservice** 表现不一致：paymentservice-0（aiops-k8s-08）rrt 99分位数从 2874.8 升至 3602.27，而 paymentservice-2（aiops-k8s-04）反而下降，说明问题非服务全局，可能与所在节点状态相关。
- **adservice** 三个 Pod 的请求量（`pod_network_receive_bytes`）均大幅下降（如 adservice-2 从 515.62 → 107.29），且 `request` 指标99分位数降至固定值 48.0，反映上游调用骤减，属服务级流量模式变化。

#### 3. Pod级别现象观察

部分 Pod 表现出孤立异常，且与所在节点高度相关：

- **shippingservice-0**（部署于 aiops-k8s-01）rrt 99分位数从 1867.95 激增至 5335.73，而同服务的 shippingservice-1（aiops-k8s-07）和 shippingservice-2（aiops-k8s-03）无显著延迟上升，表明问题局限于该 Pod。其所在节点 aiops-k8s-01 内存压力减轻，排除节点资源瓶颈，可能为 Pod 内部状态异常。
- **cartservice-2**（部署于 aiops-k8s-08）在故障期间出现 timeout 99分位数为 1.0（正常为 0.0），而 cartservice-0（aiops-k8s-07）和 cartservice-1（aiops-k8s-05）无此现象，且其 `pod_memory_working_set_bytes` 99分位数从 1.53M 升至 1.95M，略高于其他实例，可能因内存或 GC 导致偶发超时。
- **checkoutservice-2**（aiops-k8s-05）维持高延迟（rrt 99分位数约 8500ms），但未进一步恶化，其 `pod_network_*` 指标稳定，所在节点内存使用率下降，表明其高延迟为固有状态而非新发故障。
- 异常 Pod 分布广泛：shippingservice-0（aiops-k8s-01）、cartservice-2（aiops-k8s-08）、checkoutservice-2（aiops-k8s-05）分别位于不同节点，未集中于单一故障节点，说明多数异常为 Pod 或服务逻辑层面，非节点基础设施全局失效。

#### 综合现象描述

系统在故障期间呈现“局部高延迟 + 部分服务流量骤降”的复合现象。Node 层面，aiops-k8s-04 和 aiops-k8s-07 出现显著内存压力（`node_memory_usage_rate` 上升、`node_memory_MemAvailable_bytes` 下降），其中 aiops-k8s-04 还伴随磁盘写延迟（`node_disk_write_time_seconds_total`）激增。Service 层面，emailservice 和 currencyservice 普遍出现尾部延迟恶化，adservice 请求量锐减。Pod 层面，shippingservice-0 和 cartservice-2 表现孤立异常，分别体现高延迟和超时，但分布于不同节点，表明问题根源更可能在应用层或调度配置，而非底层基础设施大面积故障。整体来看，异常现象以服务级和 Pod 级为主，节点级异常仅限个别高负载节点。
