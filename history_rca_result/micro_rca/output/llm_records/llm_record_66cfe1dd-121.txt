### 系统指标综合分析结果:
### 全局综合现象总结

#### 1. Node级别现象观察  
多个节点在故障期间表现出显著的 **kpi_key** 指标变化，主要集中在计算与存储资源维度。  
- **aiops-k8s-06** 节点的 `node_cpu_usage_rate` 中位数从 34.1% 升至 41.2%，99分位数从 42.21% 升至 52.87%，是所有节点中 CPU 使用率增幅最显著的，且该节点部署了 `checkoutservice-2` 和 `emailservice-2`，均在服务层出现严重延迟或错误。  
- **aiops-k8s-04** 节点的 `node_memory_usage_rate` 中位数从 31.68% 上升至 43.13%，同时 `node_memory_MemAvailable_bytes` 中位数下降约 1.9GB，反映内存压力显著增加，而该节点承载了 `frontend-2` 和 `checkoutservice-0`，其中 `frontend` 服务整体延迟上升。  
- **aiops-k8s-03** 和 **aiops-k8s-08** 节点的 `node_disk_write_time_seconds_total` 99分位数分别从 0.06s 和 0.03s 增至 0.45s 和 0.32s，表明磁盘写入延迟在极端情况下明显恶化，可能影响 I/O 密集型 Pod。  
- **aiops-k8s-01** 节点的 `node_disk_read_bytes_total` 99分位数从 117.42 增至 1294.34，增幅超10倍，但中位数仍为 0.0，说明仅在极少数时间点出现突发读取流量。  
- **aiops-k8s-07** 节点的 `node_memory_usage_rate` 高达 65.64%（中位数），为所有节点中最高，且四分位距缩小，表明内存使用趋于饱和且波动降低。

不同节点间差异明显：高负载节点（如 aiops-k8s-06、aiops-k8s-07）集中承载了异常服务，而空载节点（如 aiops-k8s-02）指标几乎无变化。

#### 2. Service级别现象观察  
多个服务实例呈现**一致性的异常模式**，表明问题具有服务级共性。  
- **emailservice** 的三个 Pod（-0、-1、-2）均出现 RRT 中位数 11~18 倍激增，且其所在节点（aiops-k8s-07、aiops-k8s-01、aiops-k8s-06）上的 Pod 均显示 `pod_processes` 从中位数 1.0 骤增至 12.0，`pod_cpu_usage` 从 0.0 升至 0.4，表明服务内部进程数和 CPU 消耗同步异常增长，具有强一致性。  
- **checkoutservice** 虽仅有一个 Pod（-2），但其 error_ratio 99分位达 0.93，且所在节点 aiops-k8s-06 的 `node_cpu_usage_rate` 显著上升，同时 `pod_network_transmit_bytes` 99分位从 13417.27 增至 15630.17，反映请求处理失败伴随网络输出增加。  
- **frontend** 三个 Pod 均延迟上升，且分别位于 aiops-k8s-03、aiops-k8s-04、aiops-k8s-07，这些节点均存在内存或 CPU 压力，表明前端延迟可能受下游依赖及宿主资源双重影响。  
- **adservice** 三个 Pod 中仅 `adservice-0`（位于 aiops-k8s-03）报告 error_ratio 99分位达 0.5，其余 Pod 无异常，说明该服务异常不具备全局性，可能为局部问题。  
- **cartservice** 与 **redis-cart** 的 timeout 次数增加，且 `redis-cart-0` 所在节点 aiops-k8s-08 的 `node_disk_write_time_seconds_total` 99分位显著上升，暗示缓存 I/O 延迟可能影响购物车服务。

#### 3. Pod级别现象观察  
异常 Pod 呈现**跨节点分布但集中于特定服务**，未发现单一 Pod 独立异常。  
- 所有 `emailservice` Pod 均表现出 `pod_processes` 从 1 增至 12、`pod_cpu_usage` 从 0 升至 0.4，且 `pod_memory_working_set_bytes` 99分位数大幅上升（如 `emailservice-2` 从 128.42 增至 1185634.82），表明该服务在故障期间普遍触发了资源密集型行为。  
- `checkoutservice-2` 的 `pod_network_receive_bytes` 和 `pod_network_transmit_bytes` 99分位均显著上升，与其高错误率并存，可能反映重试或异常响应膨胀。  
- 异常 Pod 并非集中于单一节点：`emailservice` 分布在 aiops-k8s-01、aiops-k8s-06、aiops-k8s-07；`frontend` 分布在 aiops-k8s-03、aiops-k8s-04、aiops-k8s-07；`checkoutservice-2` 位于 aiops-k8s-06。这表明问题根源更可能来自服务自身逻辑或依赖，而非特定 Node 故障。  
- 相反，稳定服务（如 `paymentservice`、`shippingservice`）的 Pod 在各节点上指标波动极小，进一步佐证异常具有服务选择性。

#### 综合判断  
系统在故障期间呈现**以 emailservice 为核心、checkoutservice 为次级焦点的服务级性能退化**，伴随前端响应延迟和缓存超时。基础设施层面，承载异常服务的节点（尤其是 aiops-k8s-06、aiops-k8s-04、aiops-k8s-07）在 `node_cpu_usage_rate`、`node_memory_usage_rate` 和 `node_disk_write_time_seconds_total` 等 **kpi_key** 指标上出现显著压力，但无全局性资源枯竭。异常现象主要体现为**服务级别的一致性行为突变**，而非孤立 Pod 或 Node 故障，暗示问题可能源于 emailservice 内部逻辑缺陷、外部依赖瓶颈或配置异常，进而通过调用链传导至 checkoutservice 和 frontend。
