### 系统指标综合分析结果:
### 全局综合现象总结

#### 1. Node级别现象观察  
各节点在故障期间整体资源使用趋于平稳或下降，未出现明显的资源瓶颈。kpi_key指标 `node_cpu_usage_rate` 在所有节点（如 aiops-k8s-01 至 aiops-k8s-08）均呈现中位数小幅下降或基本持平，例如 aiops-k8s-03 从 30.33 降至 28.64，aiops-k8s-08 从 20.51 降至 17.42。kpi_key指标 `node_memory_usage_rate` 在高负载节点如 aiops-k8s-07（70.02 → 60.37）和 aiops-k8s-03（62.64 → 48.61）显著下降，反映上层服务请求减少导致内存压力缓解。磁盘与网络类指标如 `node_disk_written_bytes_total`、`node_network_receive_packets_total` 等普遍呈下降趋势，表明系统整体 I/O 和网络流量萎缩。值得注意的是，所有节点的 `node_disk_write_time_seconds_total` 均接近 0，无 I/O 阻塞迹象。不同节点间无显著异常差异，资源使用变化趋势一致，未发现特定节点存在资源过载或性能劣化。

#### 2. Service级别现象观察  
多个服务实例表现出高度一致的流量骤降与网络活动萎缩，体现为 service 级别的共性现象。  
- **emailservice**：三个实例（-0/-1/-2）的 `pod_network_receive_bytes` 和 `pod_network_transmit_bytes` 中位数均从百字节级降至个位数（如 emailservice-1 从 170.26 降至 5.79），同时 `pod_memory_working_set_bytes` 在故障期间中位数归零，表明服务基本停止处理请求。  
- **paymentservice**：三个实例的网络指标在故障期间中位数全部降为 0，显示完全无流量。  
- **currencyservice**：currencyservice-1 虽保留部分流量（`pod_network_receive_bytes` 从中位数 7881.49 降至 715.38），但 currencyservice-0 和 -2 的网络指标中位数归零，呈现部分实例失效特征。  
- **frontend**：三个实例（-0/-1/-2）的 `pod_network_receive_bytes` 和 `pod_network_transmit_bytes` 均大幅下降（如 frontend-2 接收字节从 21141.57 降至 5561.58），与其作为入口服务的高错误率和超时现象一致。  
- **checkoutservice** 与 **cartservice**：其 Pod 的网络流量显著减少，且伴随极端延迟，符合业务链路中断特征。  
上述共性表明，故障影响沿调用链传播，导致多个服务整体或部分实例进入低流量或无响应状态。

#### 3. Pod级别现象观察  
异常 Pod 广泛分布于多个节点，未集中于单一 Node，排除 Node 级硬件或资源问题。  
- **cartservice-2**（位于 aiops-k8s-08）：其 `pod_fs_reads_bytes` 故障期间 99 分位数突增至 61033.11，而其他 cartservice 实例无此现象，可能涉及本地 I/O 异常；同时其 `pod_network_transmit_bytes` 下降幅度小于接收端，暗示内部处理阻塞。  
- **redis-cart-0**（位于 aiops-k8s-08）：作为 cartservice 依赖，其 `pod_network_receive_bytes` 从 1556.65 降至 176.13，`pod_network_transmit_bytes` 从 724.52 降至 63.74，流量锐减，且 `pod_memory_working_set_bytes` 中位数翻倍（118万 → 265万），反映缓存层积压或响应停滞。  
- **adservice-1**（位于 aiops-k8s-07）：`pod_network_receive_bytes` 从 2320.12 骤降至 120.69，而 adservice-0 和 -2 变化较小，显示该实例孤立失效。  
- **productcatalogservice-0**（位于 aiops-k8s-08）：网络指标中位数趋近于 0，但 99 分位数仍较高（如接收字节 99 分位达 774.57），表明仅尾部请求存活，与 Service 级别观察到的局部高错误率一致。  

关键异常 Pod（如 cartservice-2、redis-cart-0、checkoutservice-0/2、frontend-1/2）分散在 aiops-k8s-01、aiops-k8s-04、aiops-k8s-05、aiops-k8s-06、aiops-k8s-08 等多个节点，无 Node 聚集性，表明问题源于应用逻辑或服务间依赖，而非基础设施局部故障。

#### 综合现象  
系统在故障期间呈现“上游服务高延迟/超时 → 中间服务流量骤降 → 下游服务无请求”的级联退化模式。Node 级资源指标（`node_cpu_usage_rate`、`node_memory_usage_rate` 等）普遍下降，反映整体负载降低，无资源争用。Service 级别上，emailservice、paymentservice 等完全静默，frontend、checkoutservice、cartservice 及 redis-cart 出现严重性能劣化。Pod 级异常广泛分布于多个节点，核心问题集中于 cartservice 与 redis-cart 的交互瓶颈，并向上游扩散至 checkoutservice 和 frontend。数据库（TiDB）及多数基础设施指标正常，进一步确认故障根因位于应用层服务依赖链。
