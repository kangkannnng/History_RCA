cartservice-0在故障期间timeout的99分位数从1.0升至9.13，而其他Pod无此现象。同时，其请求量下降超过50%（中位数从391降至169），但rrt变化不大，暗示部分请求因下游问题发生超时。

**5. 其他服务请求量普遍下降：**  
包括checkoutservice、currencyservice、shippingservice、recommendationservice等在内的多个服务，其各Pod在故障期间的request和response中位数均显著减少（普遍下降40%~70%），但rrt多数保持稳定或略有改善。这表明整体业务流量下降，可能由前端入口异常引发级联效应。

**6. 部分服务数据缺失或稳定：**  
emailservice、paymentservice、redis等服务虽有部分指标记录，但整体波动较小。例如emailservice的rrt变化幅度有限，redis的rrt 99分位数从1601ms降至670ms，timeout未增加，表现相对稳定。

### TiDB数据库组件现象

**1. TiDB层出现大量查询失败：**  
tidb-tidb组件的failed_query_ops在正常期间为0，但在故障期间中位数高达459.34，99分位数达507.6，表明数据库层面出现持续性查询失败。与此同时，CPU使用率从0.34降至0.08，内存使用量也明显下降，说明数据库负载减轻，可能因上游服务减少调用或连接中断。

**2. TiKV资源使用率下降：**  
tidb-tikv的CPU使用率从正常期间中位数0.11降至0.07，反映存储层负载降低，与上层服务请求量减少一致。

**3. PD组件数据缺失：**  
tidb-pd的所有关键指标均标记为“缺失数据”，按规则视为波动极小、运行正常，未观察到集群拓扑或健康状态的明显异常。

### 总结
系统在故障期间最突出的现象是：**前端服务出现高比例服务端错误，广告服务延迟显著增加，TiDB数据库层产生大量查询失败，同时绝大多数微服务的请求量大幅下降**。这些变化呈现明显的关联性——前端异常可能导致用户请求无法正常发起，进而引发全链路流量下降；而数据库查询失败可能进一步加剧服务异常。个别Pod（如cartservice-0）的超时异常和productcatalogservice实例间的性能分化，也值得关注。整体来看，故障对业务的影响主要体现在可用性（错误率）和响应效率（延迟）两个维度。

## 基础设施指标分类说明

- **node_filesystem_usage_rate**: 文件系统使用率, 反映节点存储空间使用率

- **node_network_transmit_packets_total**: 各接口每秒发送的数据包总数  
- **node_sockstat_TCP_inuse**: TCP连接数, 反映节点TCP连接活跃度

- **pod_network_receive_packets**: Pod 网络接收数据包数  
- **pod_network_transmit_bytes**: Pod 网络发送字节数  
- **pod_network_transmit_packets**: Pod 网络发送数据包数

        理解以下关键调用路径有助于识别故障传播和根因定位：
        
        **主要调用路径:**
        1. **用户请求入口**: User → frontend (所有用户请求的统一入口)
        2. **购物核心流程**: frontend → checkoutservice → (paymentservice, emailservice, shippingservice, currencyservice)
        3. **商品浏览相关**: frontend → (adservice, recommendationservice, productcatalogservice, cartservice)
        4. **服务间依赖**: recommendationservice → productcatalogservice (推荐依赖商品目录)
        5. **数据存储层**:
           - adservice/productcatalogservice → tidb (广告和商品数据存储)
           - cartservice → redis-cart (购物车缓存)
           - tidb 集群内部: tidb → (tidb-tidb, tidb-tikv, tidb-pd)

            **Service级别故障**: 同一服务的多个Pod实例（如emailservice-0, emailservice-1, emailservice-2）都出现相似的异常数据变化，表明服务本身存在问题
            **Pod级别故障**: 单个Pod（如cartservice-0）出现异常数据变化，而同服务的其他Pod（cartservice-1, cartservice-2）及其他Pod正常

### 系统指标综合分析结果:
### 全局现象总结

#### 1. Node级别现象观察  
各节点在故障期间的基础设施指标整体波动有限，未出现普遍性资源瓶颈。kpi_key指标`node_cpu_usage_rate`在aiops-k8s-03和aiops-k8s-04上分别从正常期间中位数45.84%、35.66%下降至37.44%、33.70%，反映计算负载降低；而aiops-k8s-05的`node_memory_usage_rate`中位数从44.99%降至31.06%，空闲内存显著增加（`node_memory_MemAvailable_bytes`中位数由8.88GB升至11.22GB），表明该节点负载明显减轻。磁盘写入活动方面，多数节点如aiops-k8s-01、aiops-k8s-03、aiops-k8s-08的`node_disk_written_bytes_total`中位数变化幅度较小（<20%），且`node_disk_write_time_seconds_total`始终接近0，说明存储I/O未成为瓶颈。网络层面，所有节点的`node_network_receive_bytes_total`与`node_network_transmit_bytes_total`均未提供完整数据，但通过Pod级网络指标可推断整体流量下降。不同节点间无一致性异常模式，资源使用率变化主要体现为负载下降，而非局部过载或故障。

#### 2. Service级别现象观察  
**Frontend服务**：所有实例（frontend-0/1/2）均表现出请求量锐减（如frontend-2的`pod_network_transmit_bytes`中位数从31998.77降至15595.40）和工作集内存下降（如frontend-1的`pod_memory_working_set_bytes`中位数从355527.58降至191.36），与Service层观测到的高错误率和低请求量一致。  
**Productcatalogservice服务**：呈现明显分化。productcatalogservice-2（部署于aiops-k8s-01）的`pod_network_receive_bytes`中位数从14755.36骤降至1704.82，而productcatalogservice-0（aiops-k8s-08）虽请求量下降（`pod_network_transmit_bytes`从6246.12降至1314.42），但其`pod_memory_working_set_bytes`中位数从4820.13降至0，反映负载极低。  
**Cartservice服务**：cartservice-0（aiops-k8s-07）的`pod_memory_working_set_bytes`中位数从718382.03暴跌至2390.46，网络流量同步下降（`pod_network_transmit_bytes`从2189.06降至937.18），而cartservice-1（aiops-k8s-05）和cartservice-2（aiops-k8s-08）的内存与网络指标相对稳定，仅小幅下降。  
**Emailservice与Paymentservice**：各实例（如emailservice-0/1/2、paymentservice-0/1/2）的`pod_network_receive_bytes`和`pod_network_transmit_bytes`普遍下降50%以上，但内存使用无显著异常，符合全局流量衰减特征。  
**Adservice服务**：adservice-0（aiops-k8s-03）和adservice-1（aiops-k8s-07）的网络流量大幅减少（如adservice-0的`pod_network_transmit_bytes`中位数从1153.20降至409.27），但adservice-2（aiops-k8s-08）的`pod_network_receive_bytes`反而从108.13微升至118.30，显示实例间行为不一致。

#### 3. Pod级别现象观察  
异常Pod分布分散，未集中于特定节点。cartservice-0（aiops-k8s-07）的`pod_memory_working_set_bytes`中位数下降超99%，远超其他cartservice实例，属孤立异常；productcatalogservice-2（aiops-k8s-01）的网络流量降幅（88%）显著高于同服务其他实例（如productcatalogservice-0降幅79%），亦属个别现象。前端Pod（frontend-0/1/2）分别部署于aiops-k8s-04、aiops-k8s-01、aiops-k8s-08，均出现请求量与内存使用同步下降，体现服务级共性而非节点级问题。TiDB组件未在Pod列表中体现，但其查询失败现象与上游服务流量下降趋势一致。整体上，绝大多数Pod的异常表现为流量和资源使用率降低，仅少数Pod（如cartservice-0、productcatalogservice-2）出现极端指标变化，且无跨Node聚集性，表明异常主要源于服务调用链中断，而非基础设施故障。

综上，系统在故障期间呈现**全局流量衰减**特征，Node资源使用率普遍下降，Service层以frontend高错误率和productcatalogservice/adservice延迟恶化为核心表现，Pod异常多为服务级共性，仅个别实例（如cartservice-0）存在极端指标波动，且无Node级资源瓶颈或故障迹象。
