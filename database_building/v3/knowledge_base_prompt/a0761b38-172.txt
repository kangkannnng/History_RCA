# Constraints (Crucial!)
1. **Data Scope Restriction**: 你的证据来源**严格限制**在 **Trace**, **Log**, **Metric** 这三种模态。即使故障是 OOM 或 节点宕机，也必须通过 Log (报错日志) 或 Metric (内存曲线/心跳断连) 来体现，**严禁捏造** K8s Event 或其他不存在的数据源。

2. **Terminology Consistency (术语一致性)**:
   - **故障组件**: 必须使用 Ground Truth 中 `instance` 字段的**完整准确名称**，不要使用简称或描述性语言
     * 例如：如果GT是 "tidb-pd-0"，不要写成 "PD组件" 或 "Placement Driver"
     * 例如：如果GT是 "shippingservice-1"，必须包含完整的pod编号
   - **故障原因**: 优先使用 Ground Truth 中 `key_metrics` 字段的术语
     * 例如：如果GT包含 "pod_cpu_usage"，在 fault_type 和 root_cause_desc 中应该提及这个指标
     * 可以适当总结，但必须保留关键术语
   - **故障特征**: critical_checks 必须覆盖 Ground Truth 中 `key_observations` 提到的模态和关键词
     * 例如：如果GT的key_observations包含 {"type": "metric", "keyword": ["pod_cpu_usage", "rrt_max"]}
     * 那么你的critical_checks中必须包含这些指标的检查项

# Processing Workflow
1. **Verdict (判卷)**:
   - 对比 Historical Conclusion 和 Ground Truth。
   - 如果结论一致，提取其正确逻辑。
   - 如果结论不一致，**忽略历史推理**，基于 Ground Truth 在 Full Context 中重新寻找 Log/Metric/Trace 的证据。

2. **Extraction (提取)**:
   - 提取能有力证明 Ground Truth 的关键线索。
   - 必须是原始数据中真实存在的（例如：不要说"Log 中有报错"，如果原始 Log 其实是干净的）。
   - **特别注意**: 确保提取的证据能够支持 Ground Truth 中指定的 key_metrics 和 key_observations

3. **Reconstruction (重构)**:
