# Constraints (Crucial!)
1. **Data Scope Restriction**: 你的证据来源**严格限制**在 **Trace**, **Log**, **Metric** 这三种模态。即使故障是 OOM 或 节点宕机，也必须通过 Log (报错日志) 或 Metric (内存曲线/心跳断连) 来体现，**严禁捏造** K8s Event 或其他不存在的数据源。

2. **Terminology Consistency (术语一致性)**:
   - **故障组件**: 必须使用 Ground Truth 中 `instance` 字段的**完整准确名称**，不要使用简称或描述性语言
     * 例如：如果GT是 "tidb-pd-0"，不要写成 "PD组件" 或 "Placement Driver"
     * 例如：如果GT是 "shippingservice-1"，必须包含完整的pod编号
   - **故障原因**: 优先使用 Ground Truth 中 `key_metrics` 字段的术语
     * 例如：如果GT包含 "pod_cpu_usage"，在 fault_type 和 root_cause_desc 中应该提及这个指标
     * 可以适当总结，但必须保留关键术语
   - **故障特征**: critical_checks 必须覆盖 Ground Truth 中 `key_observations` 提到的模态和关键词
     * 例如：如果GT的key_observations包含 {"type": "metric", "keyword": ["pod_cpu_usage", "rrt_max"]}
     * 那么你的critical_checks中必须包含这些指标的检查项

# Processing Workflow
1. **Verdict (判卷)**:
   - 对比 Historical Conclusion 和 Ground Truth。
   - 如果结论一致，提取其正确逻辑。
   - 如果结论不一致，**忽略历史推理**，基于 Ground Truth 在 Full Context 中重新寻找 Log/Metric/Trace 的证据。

2. **Extraction (提取)**:
   - 提取能有力证明 Ground Truth 的关键线索。
   - 必须是原始数据中真实存在的（例如：不要说"Log 中有报错"，如果原始 Log 其实是干净的）。
   - **特别注意**: 确保提取的证据能够支持 Ground Truth 中指定的 key_metrics 和 key_observations

3. **Reconstruction (重构)**:
   - 构建推理链条，语气必须是"通过观察数据推导出结论"，**严禁**泄露"因为 Ground Truth 说是..."的信息。
   - **在 root_cause_desc 中必须明确提及 Ground Truth 的 instance 名称**
   - **在 reasoning_chain 中应该提及 Ground Truth 的 key_metrics 相关指标**

    "expert_knowledge": {

        "reasoning_chain": [

        ],
        "critical_checks": [

# Example (示例)
假设 Ground Truth 为:
{
    "instance": ["tidb-pd-0", "tidb-pd"],
    "key_metrics": ["abnormal_region_count", "region_health"],
    "key_observations": [
        {"type": "metric", "keyword": ["abnormal_region_count", "region_health"]},
        {"type": "log", "keyword": ["error", "crash"]}
    ],
    "fault_description": ["pod crash", "pod failure"]
}

则输出应该类似:
{
    "uuid": "xxx",
    "fault_type": "abnormal_region_count",
    "symptom_vector": "pod crash with abnormal region count spike",
    "expert_knowledge": {
        "root_cause_desc": "tidb-pd-0 组件因 abnormal_region_count 异常飙升导致崩溃",
        "reasoning_chain": [
            "Step 1: 观察到 tidb-pd-0 的 abnormal_region_count 指标从正常值突然飙升至异常水平",
            "Step 2: 分析 tidb-pd 组件的 region_health 指标同时恶化",
            "Step 3: 日志中发现 tidb-pd-0 的崩溃记录，确认为 pod failure"
        ],
        "critical_checks": [
            {
                "modality": "Metric",
                "target": "abnormal_region_count",
                "expected_pattern": "Sudden spike from baseline",
                "instruction": "检查 tidb-pd 组件的 abnormal_region_count 指标是否异常飙升"
            },
            {
                "modality": "Metric",
                "target": "region_health",
                "expected_pattern": "Health score degradation",
                "instruction": "检查 region_health 指标是否恶化"
            },
            {
                "modality": "Log",
                "target": "error|crash",
                "expected_pattern": "Contains crash or error keywords",
                "instruction": "在 tidb-pd-0 的日志中搜索 crash 或 error 关键词"
            }
        ]
    }
}

**重要提醒**:
- root_cause_desc 中必须出现 "tidb-pd-0" 或 "tidb-pd" 的完整名称
- fault_type 使用了 GT 的第一个 key_metric: "abnormal_region_count"
- critical_checks 覆盖了 GT 的所有 key_observations (Metric 和 Log)
- 所有 target 都使用了 GT 中的原始 keyword
